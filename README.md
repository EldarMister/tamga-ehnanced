# Tamga CRM

Веб-приложение для полиграфии и внутреннего управления производством. Проект объединяет CRM для заказов, прайс-лист, склад, HR-модуль, задачи, обучение, объявления, расчёты и PWA-режим для работы с телефона.

## Что это за проект

`tamga-react` — это SPA на React + Vite с backend на Express + PostgreSQL. Приложение рассчитано на ежедневную работу команды полиграфии:

- менеджеры создают и ведут заказы;
- дизайнеры получают заказы в дизайн;
- мастера и помощники двигают производство;
- директор управляет сотрудниками, зарплатами, справочниками и настройками;
- сотрудники получают push-уведомления и realtime-обновления.

Фронтенд и backend могут работать:

1. раздельно в локальной разработке;
2. как один unified-сервис в проде, когда Express раздаёт собранный `dist/`.

## Основные возможности

- список и карточка заказов;
- создание заказа с позициями, расчётом стоимости и предоплатой;
- статусы заказов и история переходов;
- загрузка фото и макета;
- склад материалов и резервирование материалов под заказ;
- прайс-лист услуг и связанные материалы;
- расчёт зарплаты, штрафов и журнал работы;
- кадровый модуль, посещаемость и чек-листы смены;
- внутренние объявления;
- обучающие материалы;
- профиль пользователя, язык интерфейса и push-уведомления;
- PWA-режим с service worker;
- realtime через Server-Sent Events.

## Роли в системе

Поддерживаемые роли определены в БД и backend:

- `director`
- `manager`
- `designer`
- `master`
- `assistant`

Важные различия по правам:

- директор управляет пользователями, может редактировать свой профиль, видеть больше внутренних данных и администрировать систему;
- менеджер создаёт заказы и закрывает клиентский цикл;
- дизайнер работает со статусами дизайна;
- мастер и помощник работают с производственными стадиями;
- видимость некоторых заказов и действий фильтруется по роли прямо на backend.

- React 18

- Vite 5

## Быстрый старт

### Требования

- Node.js 18+;
- npm 9+;
- PostgreSQL;
- доступ к переменным окружения для backend.

### 1. Установка зависимостей

В корне проекта:

```bash
npm install
```

Для backend:

```bash
npm --prefix server install
```

### 2. Настройка переменных окружения

Проект использует переменные как для фронта, так и для backend.

#### Переменные frontend / Vite

Обычно размещаются в корневом `.env`.

```env
# Оставьте пустым при unified-deploy, когда фронт и API на одном origin
VITE_API_BASE=

# Используется dev-proxy Vite
TAMGA_BACKEND=http://localhost:8000
```

#### Переменные backend

Их можно задать через shell, Railway Variables или через отдельный `.env` для запуска backend.

Обязательные:

```env
DATABASE_URL=postgres://USER:PASSWORD@HOST:5432/DBNAME
SECRET_KEY=change-me-to-a-long-random-secret
```

Опциональные:

```env
JWT_EXPIRY_HOURS=720
CORS_ORIGINS=http://localhost:5173
PORT=8000
```

#### Переменные для Web Push

Push-уведомления опциональны. Если не настроены, приложение продолжит работать без падения сервера.

```env
VAPID_SUBJECT=mailto:admin@example.com
VAPID_PUBLIC_KEY=
VAPID_PRIVATE_KEY=
```

Примечания:

- `VAPID_SUBJECT` поддерживает формат `mailto:...`;
- также допустим обычный email, приложение нормализует его само;
- ключи можно сгенерировать командой:

```bash
npx web-push generate-vapid-keys
```

### 3. Запуск backend

```bash
npm --prefix server run dev
```

Backend стартует на `http://localhost:8000`.

Во время старта:

- читается схема из `server/schema.sql`;
- выполняется инициализация БД;
- при пустой БД создаются seed-данные;
- если пользователей нет, создаётся директор по умолчанию.

### 4. Запуск frontend

Во втором терминале:

```bash
npm run dev
```

Frontend стартует на `http://localhost:5173`.

Vite автоматически проксирует `/api/*` на `TAMGA_BACKEND`.

### 5. Первый вход

Если база была пустой, backend создаст стартового пользователя:

```text
Логин: admin
Пароль: 12345
```

После первого входа пароль нужно сменить.

## Что инициализируется автоматически

При `initDb()` backend:

- применяет `server/schema.sql`;
- добавляет недостающие колонки миграционным способом;
- сидит чек-листы смены;
- сидит услуги и материалы;
- сидит карту `услуга -> материал`;
- создаёт стартового пользователя `admin / 12345`, если БД пустая.

Это удобно для быстрого развёртывания, но в production важно:

- сменить стартовый пароль;
- задать свой `SECRET_KEY`;
- использовать реальную PostgreSQL-базу, а не временную.

## Авторизация и безопасность

### JWT

Токены создаются в `server/auth.js` и подписываются через `HMAC-SHA256`.

Особенности:

- секрет берётся из `SECRET_KEY`;
- срок жизни задаётся `JWT_EXPIRY_HOURS`;
- middleware `authRequired` подтягивает пользователя из БД;
- middleware `roleRequired(...roles)` ограничивает доступ по ролям.

### Хеширование паролей

Пароль хешируется как:

```text
sha256(password + SECRET_KEY)
```

Это сделано для совместимости со старой схемой проекта.

### Что обязательно сделать в production

- заменить `SECRET_KEY`;
- сменить `admin / 12345`;
- ограничить `CORS_ORIGINS`, если фронт и backend разнесены;
- хранить production-переменные только в защищённом env-хранилище.

## Orders: как устроен поток заказа

Backend в `server/routes/orders.js` содержит основную бизнес-логику заказа:

- генерация номера заказа вида `POL-YYYY-XXX`;
- расчёт суммы по услугам;
- учёт предоплаты;
- привязка и резервирование материалов;
- история смены статусов;
- ограничения переходов статусов по ролям;
- загрузка фото и дизайн-макета;
- серверная фильтрация списка и role-based visibility.

Поддерживаемые статусы:

- `created`
- `design`
- `design_done`
- `production`
- `printed`
- `postprocess`
- `ready`
- `closed`
- `cancelled`
- `defect`

Важно: UI можно редизайнить и улучшать, но backend-контракты и transitions лучше не ломать без осознанной миграции.

## Realtime

Realtime построен на Server-Sent Events:

- endpoint: `/api/events`;
- реализация: `server/realtime.js`;
- клиент подписывается с токеном;
- сервер рассылает события всем или конкретному пользователю.

Это используется для лёгких онлайн-обновлений без WebSocket-слоя.

## Push-уведомления

Push-сценарий состоит из двух частей:

1. backend хранит подписки и отдаёт публичный VAPID-ключ;
2. frontend регистрирует подписку через service worker.

Если VAPID не настроен:

- API `/api/push/config` вернёт `enabled: false`;
- UI покажет, что push недоступен;
- сервер не должен падать.

## PWA и service worker

Проект настроен как PWA через `vite-plugin-pwa` в режиме `injectManifest`.

Основные файлы:

- `vite.config.js` — конфигурация PWA;
- `src/sw.js` — service worker;
- `src/main.jsx` — регистрация `Workbox`.

Текущее поведение обновлений:

- сервис-воркер регистрируется вручную через `Workbox`;
- используется `updateViaCache: 'none'`, чтобы мобильные браузеры не держали старый `sw.js`;
- waiting-версия автоматически получает `SKIP_WAITING`;
- при смене controlling service worker страница перезагружается;
- сервер отдаёт `index.html`, `sw.js` и `manifest.webmanifest` с `no-store/no-cache`.

Именно это нужно для сценария, когда пользователи раньше были вынуждены очищать кэш браузера вручную.

## Деплой на Railway

Проект хорошо подходит под unified-deploy одним сервисом.

### Рекомендуемая схема

1. Railway билдит проект через:

```bash
npm run build
```

2. Railway запускает backend:

```bash
npm run start
```

3. Express:

- обслуживает `/api/*`;
- раздаёт статику из `dist/`;
- отдаёт `index.html` как SPA fallback.

### Переменные Railway

Минимум:

```env
DATABASE_URL=...
SECRET_KEY=...
PORT=...
```

Опционально:

```env
CORS_ORIGINS=
JWT_EXPIRY_HOURS=720
VAPID_SUBJECT=mailto:...
VAPID_PUBLIC_KEY=...
VAPID_PRIVATE_KEY=...
```

Для unified-deploy:

- `VITE_API_BASE` обычно не нужен;
- фронт работает по same-origin;
- `CORS_ORIGINS` можно оставить пустым.

## Подсказки по разработке

### Если нужно быстро изменить UI

- сначала смотри `src/pages/`;
- затем page-scoped стили в `src/app.css`;
- не меняй API-контракты без необходимости;
- если задача только про редизайн, приоритет у JSX-композиции и CSS.

### Если нужно изменить бизнес-логику заказов

- начинать с `server/routes/orders.js`;
- проверить влияние на складские резервы и историю статусов;
- проверить видимость заказов для ролей;
- не ломать payload, который уже использует `OrderDetail` и `Orders`.

### Если нужно изменить авторизацию

- смотреть `server/auth.js`;
- помнить про совместимость текущей схемы пароля;
- не менять `SECRET_KEY` на лету в production без понимания последствий для активных токенов.

## Troubleshooting

### Backend падает с `DATABASE_URL не задан`

Причина: не задана переменная `DATABASE_URL`.

Что делать:

- экспортировать env в shell;
- либо добавить env-файл для backend;
- проверить, что переменная доступна процессу `node server/index.js`.

### На телефоне не подтягиваются новые версии

Проверь:

- что backend отдаёт `sw.js` и `index.html` без долгого кеша;
- что новая сборка реально задеплоена;
- что service worker регистрируется из `src/main.jsx`;
- что браузер не держит старую вкладку сутками без перезагрузки.

Текущая версия проекта уже содержит фиксы против такого поведения.

### Push не работает

Проверь:

- наличие `VAPID_PUBLIC_KEY` и `VAPID_PRIVATE_KEY`;
- корректность `VAPID_SUBJECT`;
- HTTPS в production;
- разрешения уведомлений в браузере и системе.

### Не удаётся войти `admin / 12345`

Возможные причины:

- база не пустая и стартовый пользователь не создавался;
- пароль уже был изменён;
- `SECRET_KEY` отличается от того, с которым был создан пароль.

## Что полезно знать перед доработками

- приложение мобильное по UX, но полноценно работает и на desktop;
- нижняя навигация зависит от роли;
- часть экранов уже переведена на page-scoped стили;
- backend intentionally простой, без ORM и без сложного слоя сервисов;
- проект использует Postgres, но query helper поддерживает стиль `?` для удобства миграции со старого SQL-кода.

## Проверка перед релизом

Минимальный чек:

```bash
npm run build
```

И вручную:

1. войти под нужной ролью;
2. открыть `/orders`, `/profile`, `/more`;
3. проверить создание или изменение заказа;
4. проверить загрузку фото/макета;
5. проверить, что PWA после деплоя подхватывает новую версию без ручной очистки кэша.
